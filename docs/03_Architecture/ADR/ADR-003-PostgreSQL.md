# ADR 003: Adoption of PostgreSQL for Relational Data

## Status
Approved

## Context
Shadow Engineer requires a persistent, ACID-compliant data store for managing users, organizations, subscriptions, repository metadata, and role-based access control (RBAC).

## Problem Statement
We need a relational database that is highly reliable, supports complex queries, JSON indexing, and strict tenant data isolation.

## Decision
We will use PostgreSQL as the primary relational database.

## Alternatives Considered
*   **MySQL:** Reliable, but PostgreSQL offers superior JSONB support and advanced features like Row-Level Security (RLS).
*   **MongoDB:** NoSQL provides flexibility, but lacks strict ACID compliance and makes relational data mapping (e.g., Users -> Orgs -> Repos -> Permissions) difficult and error-prone.

## Pros
*   Industry standard for enterprise SaaS.
*   Excellent JSONB support for unstructured metadata.
*   Row-Level Security (RLS) ensures strict tenant isolation at the database level.
*   Vast ecosystem for managed hosting (AWS RDS, Aurora).

## Cons
*   Requires schema migrations and strict DBA practices.
*   Scaling writes horizontally is complex compared to NoSQL databases.

## Trade-offs
We trade the schema-less flexibility of NoSQL for strict data integrity, ACID guarantees, and powerful relational querying.

## Consequences
*   We must implement schema migration tools (Flyway or Liquibase).
*   Complex domain models must be properly normalized to prevent performance bottlenecks.

## Future Considerations
As the platform scales, we may need to implement read-replicas for heavy analytical queries generated by the Analytics Module.
